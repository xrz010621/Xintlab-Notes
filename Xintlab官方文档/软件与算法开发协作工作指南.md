# 软件与算法开发协作工作指南

> 面向新人的研发流程与工具协作规范。本指南聚焦"流程与职责"；任务管理使用 **Linear**，即时通讯与文档协作使用 **飞书**，代码托管使用 **GitHub**，其余工具以角色类别表述。

---

## 一、工具职责划分

为避免信息分散，团队按职责划分核心工具。新人需先理解每类工具的边界：

| 工具 | 主要职责 |
|---|---|
| **GitHub**（Git 托管，主） | 代码托管、版本控制、代码审查（Code Review）、模型大文件（LFS） |
| **GitLab**（Git 托管，备份） | 仓库镜像，仅用于备份与容灾 |
| **Linear**（任务 / 项目管理） | 需求拆解、任务（Issue）跟踪、优先级与排期、**决策留痕** |
| **飞书**（即时通讯 + 文档） | 日常技术讨论、快速沟通，以及承载讨论过程文档 |
| 模型训练平台 | 算法模型训练（产出大文件需专门版本管理，见第七节） |

**核心原则：代码归 GitHub，需求与决策归 Linear，过程沟通归飞书。**

---

## 二、代码仓库（Repo）管理

**仓库划分**：按**项目维度拆分仓库**，而非将所有代码集中于单一仓库。这样做的目的是便于按项目独立控制读写权限。应用框架代码与算法开发代码分属不同仓库、不共存于同一仓库，二者通过明确的接口或依赖方式联动整合，而非在代码层面混合。

**权限设置**：按仓库（即按项目）分配读写权限，确保成员只能访问其职责范围内的代码。

**通过提交关闭任务**：在 commit message 或 PR 描述中使用约定关键字（如 `Closes #编号` / `Fixes #编号`）引用对应 Issue；当提交合并（push / merge）到目标分支后，对应 Issue 会被自动关闭。注意此处自动关闭的是 **GitHub 内部的 Issue**；需求层面的任务（位于 Linear 中）需单独维护状态，见第五节。

**PR 合并规范**：每个 Pull Request 至少需要一名**非作者本人**的 Reviewer 批准（approve）后方可合并；作者不得自审自合。代码评审在 PR 中进行（见第四节）。

**仓库镜像**：在 GitLab 保存仓库镜像，**仅用于备份与容灾**，不作为日常开发协作入口。

## 三、编程风格规范

统一编程风格是大型代码库长期可维护性的基础。团队代码风格统一遵循 **Google Style Guide**（<https://google.github.io/styleguide/>）。编码前应先阅读所用语言的对应文档：

| 语言 | 风格指南 |
|---|---|
| C++ | <https://google.github.io/styleguide/cppguide.html> |
| Python | <https://google.github.io/styleguide/pyguide.html> |
| Go | <https://google.github.io/styleguide/go/> |
| Java | <https://google.github.io/styleguide/javaguide.html> |
| JavaScript | <https://google.github.io/styleguide/jsguide.html> |
| TypeScript | <https://google.github.io/styleguide/tsguide.html> |
| Shell | <https://google.github.io/styleguide/shellguide.html> |
| HTML / CSS | <https://google.github.io/styleguide/htmlcssguide.html> |
| Markdown | <https://google.github.io/styleguide/docguide/style.html> |

完整列表（含其他语言）见索引页 <https://google.github.io/styleguide/>。Google 未提供的语言，遵循该语言社区公认的主流规范。

**落地要点：**

- 尽量以自动化工具（formatter / linter）配置对应风格，在本地与 CI 中统一执行，减少人工争论。
- 风格一致性在 Code Review 中检查（见第四节）；格式类问题应由工具拦截，评审聚焦设计与逻辑。
- 提交信息（commit message）写法可参考 *How to Write a Git Commit Message*（<https://cbea.ms/git-commit/>）。

## 四、协作与评论规范

不同类型的讨论应发生在不同位置，避免信息错位：

| 讨论类型 | 应在何处 |
|---|---|
| 代码级评论（具体某行 / 某段实现） | GitHub 的 PR（Code Review） |
| 需求讨论 | Linear 的 Issue 下 |
| 日常技术讨论 | 飞书 |

注意：GitHub 的 Issue 会随 PR 合并而关闭，因此**需求讨论不要依赖 GitHub Issue**，应放在 Linear 中长期留存。

### 实操规则

**R1 — 飞书 → Linear 收敛规则**
飞书群里的技术讨论一旦超过 **10 条来回**，必须指定一人负责"结论上链"：把结论压缩成 1–2 段贴到对应 Linear Issue 的评论中，并附上飞书 thread 链接。

**R2 — Linear Issue 评论上限**
单条 Issue 评论超过 **5 轮**后，转移到飞书 thread 继续讨论；讨论结束后按 R1 收敛回 Linear。

**R3 — 决策必须有书面留痕**
任何影响**范围 / 估时 / 优先级**的决策，其最终态必须存在于 Linear 的 Issue 评论中。飞书可承载讨论过程，但**不能作为决策的最终归属地**。

## 五、需求拆解与任务（Issue）管理

**从 PRD 到 Issue**：根据 PRD 拆分具体 Linear Issue，**Issue ID 与 PRD 任务 ID 保持对应关系**，确保每个任务可追溯到其需求来源。

**交付节奏**：每个 Issue **每周必须有可交付物（deliverable）**。

### Issue 评审日（每个 Cycle 第一天）

每个 Linear Cycle 开始的**第一天为 Issue 评审日**，是本 Cycle 唯一可调整 Issue 的窗口。当天完成三件事：

1. **评审本 Cycle Issue 合理性**：检查所有 Issue 的范围、估时、依赖与可交付性，并作出必要修改（增删、改写、调整优先级与排期）。
2. **难度识别与拆分**：对自己负责的 Issue 及其上游依赖 Issue 中过于复杂、一周内无法交付的任务，进行难度识别并拆成可在一周内交付的更小 Issue。
3. **项目级卡点评估**：评估整个项目当前卡点，判断是否需在范围、排期或资源上作出适当调整（详见第六节）。

**冻结规则**：评审日结束后，本 Cycle 内已有 Issue **冻结——不可删除、不可修改**；一经冻结，对其的意见与实时反馈**只能通过 Comment 提出**。但**允许新增 Issue** 以补充需求——冻结约束的是既有 Issue 的改动，不限制新需求的提出。

---

## 六、进度管理与卡点处理

当 Issue 出现延期或未按时完成时，由 **PM** 负责识别卡点并调整计划，包括重新评估难度、拆分任务、调整优先级或排期。

项目级卡点的常规评估时点是每个 Cycle 的 **Issue 评审日**（见第五节）：当天除评审 Issue 外，还须评估整个项目卡点，判断是否需在范围、排期或资源上作出调整。评审日的早期难度识别与拆分是预防卡点的主要手段；评审日之外暴露的严重卡点，由 PM 即时介入处理。

## 七、模型与大文件版本管理

ML 项目产生两类大文件，管理需求不同，须分别处理。**总体决策：模型文件使用 GitHub LFS（与代码 commit 绑定、可回滚）；训练数据 / GT 验证数据使用 GPU 服务器本地存储 + NAS 备份（只备份、不做版本追踪）。**

核心矛盾：Git 不适合管理大文件（>100 MB 会导致仓库膨胀、clone 缓慢），但模型文件需与代码版本绑定追踪；训练数据体量过大，版本追踪意义不大。两类文件需要不同的管理策略。

### 7.1 文件分类与管理策略

| 文件类型 | 典型大小 | 核心需求 | 策略 |
|---|---|---|---|
| 模型文件（`.mlpackage` / `.pt` / `.onnx` 等） | 20–200 MB/个 | 版本追踪、与 commit 绑定、可回滚 | GitHub LFS |
| 训练数据集（图像 / 视频 / 标注） | GB–TB 级 | 安全备份，不需版本追踪 | 本地存储 + NAS 备份 |
| GT 验证数据（基准测试数据） | GB 级 | 安全备份，不需版本追踪 | 本地存储 + NAS 备份 |
| 标注文件（YOLO TXT / COCO JSON） | <100 MB | 随代码版本管理 | 随代码正常 commit |
| 精度报告 / 截图 | <10 MB | 随 Issue 或 commit | Linear 附件或随 commit |

### 7.2 方案比较

| 维度 | GitHub LFS | DVC + S3 | 网盘（飞书 / Google Drive） |
|---|---|---|---|
| 上手成本 | 低（一次配置，git 命令不变） | 中（需学 dvc 命令集） | 低（手动上传下载） |
| 版本追踪 | 自动，与 commit 绑定 | 自动，`.dvc` 指针文件 | 手动命名区分 |
| 团队协作 | `git pull` 自动同步 | `dvc pull` 同步 | 手动共享链接 |
| 与 GitHub 联动 | 原生，PR 自动关联 | 需配置 CI | 不支持 |
| 存储成本 | 免费 1 GB，超出约 $5 / 50 GB | S3 约 $0.023 / GB / 月 | 免费额度较大 |
| 适合体量 | <1 GB 模型文件 | 任意大小 | 任意大小 |
| 适合数据集 | 否（不适合 GB 级数据） | 是 | 是 |

模型文件场景下，GitHub LFS 综合最优。

### 7.3 决策原因

**模型文件选 GitHub LFS**：git 命令不变、零学习成本；版本与 commit 自动绑定、可在任意历史节点还原对应模型；PR 合并自动联动 Linear Issue、无需手动维护状态；单文件 20–200 MB 远低于 LFS 免费额度（1 GB），现阶段零成本；权限即仓库权限，不引入额外权限系统。

**为什么不选 DVC + S3**：DVC 功能更强，但需额外维护 S3 账号、IAM 权限与 dvc 命令集。在模型文件体量未超过 LFS 免费额度的阶段，这部分复杂度没有对应收益。当模型文件总量超过 500 MB 时，重新评估是否迁移 DVC。

**训练 / GT 数据选本地 + NAS**：体量 GB–TB 级，LFS / DVC 均需付费且性价比低；原始采集数据不修改，只需备份保全、不需多版本回溯；训练需高速读取大量图像，本地 GPU 服务器访问远优于云端；本地 + NAS 异地备份满足数据安全要求。

### 7.4 GitHub LFS 操作指南

**一次性初始化**（仓库根目录；LFS 每台机器装一次，`.gitattributes` 须提交进 Git）：

```bash
git lfs install
git lfs track '*.mlpackage' '*.mlmodelc' '*.pt' '*.pth' '*.onnx' '*.bin'
git add .gitattributes
git commit -m 'chore: configure GitHub LFS for model files'
git push
```

配置完成后 `.gitattributes` 记录 LFS 追踪规则，团队成员克隆后自动应用，无需重复配置。

**日常工作流**：模型文件操作与普通文件相同——`git add / commit / push` 由 LFS 自动处理，其他成员 `git pull` 即自动下载模型文件。

**验证 LFS 生效**：`git lfs ls-files` 查看被 LFS 追踪的文件列表，`git lfs env` 查看存储使用情况。

**PR 规范**：模型文件更新的 PR 描述必须写明模型版本、关键指标（延迟 / 精度）、对应 Linear Issue 编号。

**新成员接入**：克隆后执行 `git lfs pull` 拉取模型文件；若未安装 git-lfs，需先安装（macOS `brew install git-lfs`，Linux `apt install git-lfs` 或 `conda install -c conda-forge git-lfs`）。

### 7.5 训练 / GT 数据本地存储规范

目录命名格式：`{任务类型}_{版本}_{日期}_{数量}_{简要描述}/`，示例 `real_capture_v1_20260512_300imgs_ball_detection/`。

规则：每次采集完成当天同步备份至 NAS（NAS 只保留最新版本以节省空间）；原始采集数据（图像 / 视频原文件）永久保留、不删除；数据集变更时在共享文档（飞书）更新目录记录。

### 7.6 Linear 附件挂载规范

| 内容类型 | 挂载方式 | 说明 |
|---|---|---|
| 模型文件更新 | GitHub PR 链接 | 在 Linear Issue 评论贴 PR 链接，附关键指标 |
| 精度报告截图（<10 MB） | 直接上传 Linear | RMSE 报告图表等 |
| 训练数据集引用 | 共享文档链接 + 本地路径 | 注明数据集版本、数量、采集日期 |
| GT 验证数据引用 | 共享文档链接 + 本地路径 | 注明验证样本量、采集条件 |

### 7.7 注意事项

- `.gitattributes` 必须进 Git，不可在 `.gitignore` 中排除。
- 不要直接 `git add` 大型数据集目录，确保数据集路径已在 `.gitignore` 中；一旦以非 LFS 方式提交了 >100 MB 文件，需用 `git filter-repo` 从历史中彻底清除，操作复杂——配置好 `.gitattributes` 后提交可避免该问题。
- LFS 文件删除后仍占 LFS 配额，定期 `git lfs prune` 清理本地缓存。
- **配额监控**：免费额度为 1 GB 存储 + 1 GB 带宽 / 月。模型文件总量超过 500 MB 时重新评估是否迁移 DVC + S3；接近 800 MB（逼近免费额度）时决定升级付费或迁移。每月初检查一次用量（GitHub 仓库 Settings → Billing → Git LFS Data）。
