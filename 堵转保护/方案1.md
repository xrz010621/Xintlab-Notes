## 针对角度堵转判定窗口过大的修改方案
>相关issue：[堵转保护中霍尔传感器固件可能缺失+多次堵转后阻尼器失效·Issue #42·xintlabs/puttreal-firmware-apm32](https://github.com/xintlabs/puttreal-firmware-apm32/issues/42#issue-4742349703)

#### 问题背景
摇臂多次堵转（约 10–20 次）使阻尼器（滑动离合）退化后，电机长时间空转。

#### 根因分析
仅针对**角度堵转又慢又易被绕过**部分 — `motor_controller.c` 的 3 秒窗口 + 锚点重置机制
**代码位置**：[motor_controller.c:316-350]
```c
// L316-326: 净进展角度堵转判定
if (!ctrl->is_startup_phase && (fabsf(pos_err) > POSITION_TOLERANCE))
// 不在启动阶段且距离目标位置较远(即不在死区内)
{
    if (fabsf(now_angle - ctrl->stall_anchor_angle) >= STALL_PROGRESS_DEG) // 阈值
    {
        ctrl->stall_anchor_angle = now_angle;  // 窗口内进展 ≥0.5° → 重锚，清零
        ctrl->stall_counter = 0U;
    }
    else
    {
        ctrl->stall_counter++;  // ← 每个 CONTROL_LOOP_MS(10ms) +1
    }
}
....
// L345: stall_counter > 300 → 判定堵转 (300 × 10ms = 3s)
else if (ctrl->stall_counter > STALL_DETECT_CYCLES)
    {
        /* No net headway for the whole window while off-target — a real jam. */
        motor_ctrl_enter_fault(ctrl, now_angle, MOTOR_FAULT_STALL);
        state_changed = true;
    }
```
可以发现角度堵转的判定为两个部分：**角度净进展<0.5°**且**时间长于300个周期(3s)**。并且由于计数机制和锚点设置，只需要**在3s内角度进展≥0.5°**即可消除该堵转计数。
考虑两种场景下净进展角度堵转判定出现的问题：

| 场景                             | 净进展角度堵转     |
| ------------------------------ | ----------- |
| 1 阻尼器退化+摇臂完全堵住（冻结+0.35A）       | 3.0 s 触发（慢） |
| 2 阻尼器退化+摇臂缓慢蠕动（≥0.5°/3s+0.35A） | 永不触发 ❌      |

#### 其他保护机制分析
考虑其他方面可能进行的保护，分为以下几个考虑方向：
##### **1 跟随误差极限保护 

代码第一行 `pos_err = ctrl->target_angle - now_angle;`。
- 如果上位机下发的指令是让电机以 10°/s 的速度移动，而电机因为阻尼器故障只能以 0.2°/s 蠕动，那么 `pos_err` 会以每秒 9.8° 的速度疯狂累积。
- 在稍微外层一点的控制逻辑中，如果有一个判定：`if (fabsf(pos_err) > MAX_ALLOWED_ERROR)`（例如误差超过 5° 或 10°）。一旦触发，系统直接报“位置跟随超差”故障并停机，根本等不到 3 秒去判定堵转。

**结论**：这部分缺失，`pos_err` 只用来判断是否进入 1.5° 的目标死区，并没有做最大允许误差的跳闸保护。
##### 2 全局轨迹超时保护 

即使系统允许极大的位置误差，轨迹规划器也会设定一个总时间。例如，一段 30° 的行程，正常情况只需 1 秒。如果电机以 0.2°/s 的速度蠕动，需要 150 秒才能走完。外层状态机如果有一个定时器，规定“本次动作最长允许 5 秒，超时未到达即报 **Timeout Fault**”。

**结论**：系统目前只能靠 6000ms 的 `timeout` 来强行结束，并非对不同动作区分时间限定。

##### 3 持续大电流/热过载保护

如果电机试图推动一个卡死的摇臂，即使它通过机械形变维持了 0.2°/s 的蠕动，它的**输出力矩必然极大**，这意味着它必须持续抽取极高的电流。

- 代码里的 `OVERCURRENT_LIMIT_MA` 逻辑会生效。
- 或者更高级的电机控制器会计算 $I^2t$（热积聚），当电机长时间处于大电流状态时，会报“过载 (Overload)”或“过热 (Overheating)”故障。

**结论**：因为电机打滑了，负载极小，电流非常正常（甚至偏低），根本摸不到过流保护的边缘。（即issue中过流保护不会被触发）

#### 修改方向
##### 1 引入“打滑”专属状态判定
当阻尼器退化后，出现的物理特征是：**控制器正在输出，但负载既没速度，也没高电流。**

- **特征匹配**：
    1. `fabsf(pos_err)` 很大（距离目标远）。
    2. 控制器的输出（比如给到电机的 PWM 占空比 `ctrl->actual_pwm`）已经达到了最大值的较高比例（例如 > 80%），说明电机正在驱动。
    3. 电机电流 `currentMa` **处于低位水平**（证明没有硬性卡死）。
    4. 实际摇臂速度 `derivative` **极低**。
        
- **判定**：如果这四个条件同时满足并持续比如 1000ms（100个周期），可以直接报出一个新的故障码 `MOTOR_FAULT_SLIPPAGE`（传动打滑），直接停机。

- 

##### 2 跟随误差 (Following Error)保护 
当前系统的**控制模式是 PD 位置控制 + 速度前馈**，通过**梯形轨迹规划器**`motor_ctrl_update_profile`生成`ideal_angle`(每10ms更新)，通过MT6701获取`now_angle`，得到`error = ideal_angle - now_angle`。
**轨迹规划器是开环的**：`profile_angle` 和 `profile_velocity` 纯粹基于时间推进，**不接收实际位置的反馈**。这意味着即使摇臂卡住不动，`ideal_angle` 仍然每周期向前推进。
**只有一个反馈环**：MT6701 → `now_angle` → `error` → PD → PWM。即`error`仅作为PWM的计算输入，未用作反馈控制。

- **具体分析**：
	1. 不同场景对比：正常运动中 - `error`通常**很小**；阻尼器打滑、摇臂卡住时 - `error`**持续增大**（ideal_angle 每周期前进）
	2. 不同模式下对比：
		- `SUB_MODE_POSITION`：轨迹规划器运行，`ideal_angle` 每周期更新
		- `SUB_MODE_TIMED`：`target_angle` 通过 `clamp_operating_angle(angle_deg)` 设置，`is_startup_phase = true`。进入 `motor_ctrl_update` 后会调用 `motor_ctrl_update_profile`，所以轨迹规划器会运行。但 `target_angle` 的设定取决于网关传的 `angle_deg` 参数。
		- `SUB_MODE_SPEED`：不进入 `motor_ctrl_update`（L279-282 直接 return），没有轨迹规划，`ideal_angle` 不更新
- **判定**：新增参数跟随误差阈值(如5°)，超过阈值时计数，在一定周期窗(如20个周期，~200ms)内累计计数，则判断为`MOTOR_FAULT_STALL`。

- 

##### 总结：
方案二（跟随误差）作为主检测路径：
  - 放在 `motor_controller.c`，在 `motor_ctrl_update()` 的 L295 (angle_change 计算后，此时已通过`motor_ctrl_update_profile`更新轨迹角度)
  - 条件：`!is_startup_phase && fabsf(error) > FOLLOWING_ERROR_THRESH (如 5°~8°)`，阈值一般设定为正常的 `following_error` 峰值（通常在最大加速度的瞬间最大）1.5至2倍。
  - 持续 `FOLLOWING_ERROR_TRIP_CYCLES` (如 20 = 200ms) → `MOTOR_FAULT_STALL`，远小于当前的3s净进展判定，响应更快。

方案一（打滑判定）作为覆盖<u>速度模式</u>的补充：
  - 放在 `main.c RunControlLoop`，或修改 motor_ctrl_update 支持 SUB_MODE_SPEED
  - 条件：`PWM > 80% && 电流 < 500mA && angle_change < 0.1°`
  - 持续 `1000ms` → `MOTOR_FAULT_SLIPPAGE (新增故障码)`

保留现有的角度堵转（3s 净进展）作为基础堵转判定。
