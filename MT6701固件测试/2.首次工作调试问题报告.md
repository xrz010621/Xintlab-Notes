## 问题一：Keil 工程首次编译失败

### 【问题现象】

在 Keil MDK V5.40 中打开工程后，执行 Build (F7) 依次出现以下编译/链接错误：

1. **编译器版本不可用**
   ```
   Error: Target 'DC_Motor' uses ARM-Compiler 'Default Compiler Version 5'
   which is not available.
   ```

2. **芯片系列宏未定义**
   ```
   apm32f10x.h(110): error: #35: #error directive:
   "Please select first the target APM32F10x device used in your application"
   ```

3. **公共头文件路径缺失**
   ```
   main.c(21): error: #5: cannot open source input file "ota_layout.h"
   board_driver.c(8): error: #5: cannot open source input file "ota_protocol.h"
   ```

4. **源文件未加入工程 — 链接阶段未定义符号**
   ```
   Undefined symbol NodeIdGet / NodeIdInit / NodeIdSet / NodeIdGetUid
   Undefined symbol Crc32Compute
   Undefined symbol SensorErrorAngleIsValid / SensorErrorReportUpdate / SensorErrorCountUpdate
   Undefined symbol TestShellInit / TestShellPoll
   ```

### 【形成原因】

| 子问题    | 根因                                                                                                                                                                                                         |
| ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 编译器不可用 | Keil V5.40 默认仅安装 ARM Compiler V6 (ARMCLANG)，而该工程创建时使用的是 ARM Compiler V5 (ARMCC)。V5 编译器已从新版 Keil 中移除，需单独下载安装                                                                                                |
| 宏未定义   | Geehy 标准库头文件 `apm32f10x.h` 要求预定义芯片系列宏（如 `APM32F103xB`），但工程仅定义了具体型号宏 `APM32F103CB`。系列宏用于选择正确的子头文件和寄存器映射                                                                                                     |
| 路径缺失   | 工程中 `main.c` 和 `board_driver.c` 引用了 OTA 协议和布局相关的公共头文件 (`ota_layout.h`, `ota_protocol.h`)，它们位于 `motor_control_apm32\common\` 目录下，但该目录未被加入 Keil 工程的 Include Paths                                            |
| 源文件缺失  | 以下 `.c` 源文件存在于 `app\Source\` 目录但未被加入 Keil 工程：<br>• `node_id.c` — CAN 节点 ID 管理<br>• `crc32.c` — Flash 校准数据 CRC 校验<br>• `sensor_error.c` — 传感器错误计数与判定<br>• `test_shell.c` — UART 命令 Shell (需 `TEST_BUILD` 宏) |

### 【调试思路】

```
编译错误
  ├─ 查看 Build Output 窗口 → 定位第一个错误
  ├─ 编译器版本 → 检查 Keil 安装目录下的 ARMCC 版本
  ├─ 宏/路径错误 → 读取对应源文件，查看 #include 和 #if 逻辑
  └─ 链接错误 → 根据 Undefined symbol 名称反向搜索源文件位置
       └─ 对比工程 Source Group 与实际目录中的 .c 文件列表
```

### 【修改方式】

| 序号  | 修改位置                                         | 操作                                                                                 |
| --- | -------------------------------------------- | ---------------------------------------------------------------------------------- |
| 1   | Keil IDE 外部                                  | 从 ARM 官网下载 ARM Compiler 5.06，安装后在 `Project → Manage → Folders/Extensions` 中注册编译器路径 |
| 2   | `Options for Target → C/C++ → Define`        | 添加 `APM32F103xB`，修改后为：`APM32F103CB, APM32F103xB`                                   |
| 3   | `Options for Target → C/C++ → Include Paths` | 添加 `..\..\common`                                                                  |
| 4   | Project 面板 → Source Group                    | 右键 Add Existing Files，添加 `node_id.c`, `crc32.c`, `sensor_error.c`                  |
| 5   | (后续) 同上                                      | 定义 `TEST_BUILD` 宏后需额外添加 `test_shell.c`                                             |

> 注：为了能成功完成条件宏定义的`\#ifdef TEST_BUILD`，在`test_shell.c`第26行新增`\#define TEST_BUILD`宏定义。（或在Keil Option - C/C++ 中增加`TEST_BUILD`定义。

### 【最终结果】

工程编译通过，链接成功，生成 `DC_Motor_APM32.axf`，可正常烧录。

## 问题二：MT6701 角度传感器 I2C 通信失败

### 【问题现象】

1. 固件烧录后上电运行，串口周期性输出中 `enc_err` 快速递增：
   ```
   enc_err 52 → 103 → 155 → 206 → 255
   ```
2. 传感器原始角度始终为 0：
   ```
   raw=0.00
   ```
3. 每 500ms 串口出现：
   ```
   MT6701 read_failed last_raw=0 err=255
   ```
4. 发送 `MOVE` 指令后电机完全不动，目标角度被重置为 0.00°。

### 【形成原因】

I2C 总线的 SDA 和 SCL 引脚在 `board_config.h` 中的定义与实际硬件接法**交叉互换了**：

```c
/* 代码定义 (board_config.h) — 错误 */
#define BOARD_I2C_SDA_PIN   GPIO_PIN_10   // PB10 = SDA
#define BOARD_I2C_SCL_PIN   GPIO_PIN_11   // PB11 = SCL

/* 实际硬件接法 */
// PB10 → SCL（时钟线）
// PB11 → SDA（数据线）
// 代码中 SDA/SCL 与硬件交叉互换了
```

导致 MCU 发出的 I2C START 条件和地址字节在时序上完全错误，MT6701 无法识别合法的 I2C 帧，每次访问均返回 NACK。

**错误计数机制分析**（[sensor_error.c](app/Source/sensor_error.c) 第 9-20 行）：

```c
uint8_t SensorErrorCountUpdate(uint8_t count, uint8_t read_ok)
{
    if (read_ok != 0U) { return 0U; }   // 一次成功读取即清零
    if (count < 255U)  { count++; }      // 每次失败加 1，饱和 255
    return count;
}
```

| 正常连接 | 当前异常 |
|----------|----------|
| 每次成功读 → `enc_err` 归零 | 每次读取都失败 |
| `enc_err` 始终为 0 | `enc_err` 52→103→155→206→255 |
| `raw` 随转动变化 | `raw` 始终为 0.00 |

### 【调试思路】

```
电机不动
  ├─ 确认固件在运行 → 串口有输出，LED 亮 ✅
  ├─ 确认收到了 MOVE 指令 → 串口显示 "MOVE dispatched" ✅
  ├─ 查看 STATUS → raw=0.00, enc_err=255
  │     └─ MT6701 传感器完全不工作 ← 定位目标
  ├─ 硬件排查
  │     ├─ 万用表量 MT6701 VCC-GND: ~3.3V ✅
  │     ├─ 万用表量 SDA/SCL 上拉: ~3.3V ✅
  │     ├─ 万用表量 PB10/PB11 电压与 SDA/SCL 一致 ✅
  │     ├─ 确认 MT6701 MODE 引脚电平: 3.3V (SSI 模式)
  │     │     └─ 查阅数据手册 → MT6701CT-STD-R 是 SSI 版本
  │     │         但 MODE 拉低可切换为 I2C（排除芯片版本问题）
  │     └─ 供电、上拉、接线导通性均正常
  └─ 回到代码 → 对比 board_config.h 与实际 PCB 走线
        └─ 发现 SDA/SCL 引脚定义与硬件交叉
```

### 【修改方式】

**文件**: [app/Include/board_config.h](app/Include/board_config.h) 第 15-16 行

```c
/* 修改前 */
#define BOARD_I2C_SDA_PIN   GPIO_PIN_10   // PB10 = SDA
#define BOARD_I2C_SCL_PIN   GPIO_PIN_11   // PB11 = SCL

/* 修改后 */
#define BOARD_I2C_SDA_PIN   GPIO_PIN_11   // PB11 = SDA
#define BOARD_I2C_SCL_PIN   GPIO_PIN_10   // PB10 = SCL
```

> **注**：所有 I2C 操作（`Mt6701Start`, `Mt6701WriteByte`, `Mt6701ReadByte` 等均定义在 [sensor_mt6701.c](app/Source/sensor_mt6701.c)）均通过这两个宏引用引脚，仅需修改 `board_config.h` 即可全局生效，无需改动驱动层代码。

### 【最终结果】

- `enc_err` 始终为 **0**
- `raw` 读数随电机转动正常变化（0~16383 对应 0~360°）
- `sensor_err` = 0（传感器无错误）
- 闭环位置控制恢复正常

## 问题三：IPROPI 电流检测引脚配置错误导致误触发 Stall 保护

### 【问题现象】

1. **空闲电流异常偏高**：PWM=0 时 `current_ma` 显示 880~900mA（正常应 < 100mA）
2. **发送 MOVE 后电机不动**，约 500ms 后触发保护：
   ```
   MOVE 1000
     → current_mA=977
     → Current protection trip reason=1 (MOTOR_FAULT_STALL)
     → PWM 归零，电机停止
     → angle 始终不变（0.00°），target 被重置
   ```
3. 保护触发后 `fault=1`

### 【形成原因】
#### 技术文档来源（./README.md）
- **Protection** — encoder-angle-frozen stall + motion-timeout (primary), plus a current-based overcurrent/stall backstop (IPROPI ADC). Faults latch and report over CAN.
- **保护** — 编码器角度冻结失速 + 运动超时（主要保护），外加基于电流的过电流/失速保护（`IPROPI ADC`）。故障锁定并通过CAN总线报告。

#### 硬件层

IPROPI 是电机驱动芯片的电流镜像输出引脚，输出一个与电机电流成比例的微小电流。该电流流经检流电阻 (R = 2200Ω) 到 GND，产生电压由 MCU ADC 采集。

电路图中 IPROPI 连接到 MCU 的 **PA3**，但 `board_config.h` 中错误配置为 **PA0**：

```c
/* 代码定义 (错误) */
#define BOARD_IPROPI_GPIO_PIN    GPIO_PIN_0    // PA0
#define BOARD_IPROPI_ADC_CHANNEL ADC_CHANNEL_0

/* 实际硬件 */
// IPROPI → PA3 (ADC_CHANNEL_3)
```

PA0 处于悬空状态，ADC 始终读到约 1.9V 的浮空电压，被换算为 ~880mA 的「幽灵电流」。

**电流换算公式**（[adc_service.c](app/Source/adc_service.c) 第 71-81 行）：

```
mV  = ADC_raw × 3300 / 4095
uA  = mV × 1000 / 2200        (IPROPI 镜像电流)
mA  = uA × 1000 / 1000        (电机电流 = IPROPI电流 / 增益比)
```

当 `ADC_raw = 2359` 时：`mV = 1901mV`，`uA = 864µA`，`mA = 864mA`。

#### 软件层 — Stall 保护触发链

保护逻辑位于 [control_protection.c](app/Source/control_protection.c) 第 40-90 行：

```
空闲状态:
  current_ma = 880mA > STALL_CURRENT_LIMIT_MA (800mA)
  → s_stallMs 持续累加 (每 10ms)
  → s_stall 被置 1

MOVE 指令:
  → motor_ctrl_set_target() → status = RUNNING
  → motor_ctrl_update() → 施加 PWM
  → 真实电机电流正常，但 PA3（正确引脚）无 ADC 配置，PA0 仍读到浮空电压
  → ControlProtectionUpdate(880mA, angle)
  → current >= 800mA AND angle 不变
  → s_stallMs 快速累积到 500ms
  → 触发 stall 保护 (reason=1, MOTOR_FAULT_STALL)
  → motor_ctrl_enter_fault() → PWM=0, target 重置
```

**保护阈值定义**（[app_config.h](app/Include/app_config.h)）：

| 参数 | 值 | 说明 |
|------|-----|------|
| `OVERCURRENT_LIMIT_MA` | 1000 mA | 过流阈值 |
| `OVERCURRENT_TRIP_MS` | 200 ms | 过流持续时间 |
| `STALL_CURRENT_LIMIT_MA` | 800 mA | 堵转电流阈值 |
| `STALL_CURRENT_TRIP_MS` | 500 ms | 堵转持续时间 |

862mA 未触及过流 (1000mA)，但已超过堵转阈值 (800mA)，因此触发 `MOTOR_FAULT_STALL` (reason=1)。

### 【调试思路】

```
MOVE 后电机不动
  ├─ 确认 MOVE 已被处理 → "MOVE dispatched" + target_angle 更新 ✅
  ├─ 确认 I2C/MT6701 正常 → enc_err=0, raw 有变化 ✅
  ├─ 查看 STATUS → fault=1, current_ma=899 (异常高!)
  │     └─ PWM=0 时空闲电流不应超过 100mA
  │     └─ 但 MOVE 前 current_ma 已经是 ~880mA ← 硬件信号问题
  ├─ 硬件排查
  │     ├─ 万用表量 PA0 (代码配置的 IPROPI) 电压: ~1.9V ❌
  │     │     └─ PWM=0 时 IPROPI 应输出接近 0V，1.9V 为悬空电压
  │     ├─ 万用表量 PA3 (电路图实际的 IPROPI) 电压: ~0V ✅
  │     └─ 查阅电路图 → IPROPI 实际接 PA3 而非 PA0 ← 定位根因
  └─ 修改 board_config.h 中 IPROPI 引脚定义
```

### 【修改方式】

**文件**: [app/Include/board_config.h](app/Include/board_config.h) 第 13-14 行

```c
/* 修改前 */
#define BOARD_IPROPI_GPIO_PIN    GPIO_PIN_0    // PA0
#define BOARD_IPROPI_ADC_CHANNEL ADC_CHANNEL_0

/* 修改后 */
#define BOARD_IPROPI_GPIO_PIN    GPIO_PIN_3    // PA3
#define BOARD_IPROPI_ADC_CHANNEL ADC_CHANNEL_3
```

> **注**：APM32F103CB 的 PA3 对应 ADC1 通道 3，GPIO 编号与 ADC 通道号一致。`adc_service.c` 中的 ADC 初始化代码通过 `BOARD_IPROPI_GPIO_PIN` 和 `BOARD_IPROPI_ADC_CHANNEL` 宏配置，无需额外修改。电流换算参数 (`GAIN_UA=1000`, `R_OHM=2200`, `VREF=3300`) 保持不变。

### 【最终结果】

- 空闲电流从 ~880mA 降至正常水平
- MOVE 指令正常执行，电机到达目标位置
- 不再触发 false stall 保护
- 闭环位置控制精度：误差 < 0.2°
