## 2. 系统架构简述

### 2.1 枚举协议

```
   ESP32 Master                          APM32 Slave(s)
      │                                       │
      │──── ENUM_POLL (0x7E3) ───────────────→│  roll-call
      │──── ENUM_OFFER(0x7E0, id=N) ─────────→│  offer lowest free id
      │                                       │
      │←─── ENUM_ACK (0x7E1, id, uid32) ─────│  staggered by 0-47ms UID backoff
      │                                       │
      │──── ENUM_ASSIGN(0x7E2, id, target_uid)→│  collision resolution
```

- **IDLE周期**：400ms（填充期）/ 1000ms（全部就位后）
- **COLLECTING窗口**：250ms
- **ID范围**：1~6（`NODE_ID_MIN` ~ `NODE_ID_MAX`）
- **CAN速率**：50 kbps

### 2.2 从机 CAN RX 架构

```
┌─────────────────────────────────────────────────────────┐
│ APM32F103 bxCAN 外设                                    │
│  ┌──────────────────────────────────────────┐           │
│  │ Hardware Filter (ID List Mode)           │           │
│  │  Bank 0: 0x000, 0x7DD, 0x7E0, 0x7E2    │           │
│  │  Bank 1: 0x7E3 × 4                      │           │
│  └──────────────┬───────────────────────────┘           │
│                 ▼                                       │
│  ┌──────────────────────────────────────────┐           │
│  │ RX FIFO0 (仅 3 帧深度)                    │  ← 瓶颈!  │
│  └──────────────┬───────────────────────────┘           │
│                 ▼                                       │
│  ┌──────────────────────────────────────────┐           │
│  │ 软件轮询排空 (每 10ms 一次)               │           │
│  │  CanServicePoll()       → 读 1 帧        │           │
│  │  CanServiceReceiveCommand() → 读 ≤8 帧   │           │
│  └──────────────────────────────────────────┘           │
└─────────────────────────────────────────────────────────┘
```

**关键约束**：
- FIFO 仅 3 帧深度（硬件限制）
- **不使用 CAN 接收中断**，完全依赖 10ms 周期轮询
- 10ms 内 CAN 总线可到达约 **4~5 帧**（50kbps 下每帧约 2.3ms）

## 3. 可能原因分析

### 3.1 CAN RX FIFO 溢出（主因）

#### 机理

从机的 CAN RX FIFO 只有 3 帧深度。在以下条件下会溢出：

1. **多帧突发到达**：主机在每个枚举周期开始时刻连续发送 POLL + OFFER（2帧），加上启动时的 CONFIG 广播（1帧），加上可能的电机命令帧
2. **软件排空延迟**：轮询周期 10ms，单次仅排空 1~8 帧，且中间被阻塞操作打断
3. **FIFO 溢出后果**：第4帧及之后到达的帧被硬件丢弃（FIFO overrun）

#### 证据链

**证据1**：从机侧日志显示 POLL 帧接收间隔约 650ms，而主机每 400ms 发送一次（*或是400ms+250ms，period time+Collecting window，暂定*） → 说明约 38% 的周期中帧丢失了。

**证据2**：主机 `tx_fail=0` 但 `rx=0` 持续多个cycle → 主机发送成功但收不到从机回复 → 从机根本没收到 OFFER 帧（或收到了但无法回复）。

**证据3**：从机在正常流程中每个 cycle 应收到 POLL + OFFER 共 2 帧（POLL和OFFER间隔仅10ms）。但在异常情况下仅收到 POLL → OFFER 帧被选择性丢弃，这与 FIFO 满时最后到达帧被丢弃的模式一致（主机发送顺序为 POLL → OFFER，OFFER 后到达）。

**证据4**：过程记录中的"调用情况2"显示，在从机侧添加阻塞式 UART 日志后，需要更多周期才能收到 OFFER；而不加日志时收敛更快。这与"阻塞操作增加 FIFO 溢出概率"的假设一致。

#### 溢出场景的精确时序

```
时刻     主机(ESP32)                         从机(APM32)
────────────────────────────────────────────────────────────
T+0      上电                                 上电
T+80ms   初始化中...                          CAN 外设初始化完成
T+200ms  hal_delay_ms(200) 完成               控制循环运行中
T+200ms  gateway_broadcast_config()           ← CONFIG 帧到达
         发送 CAN CONFIG 帧(0x000)            FIFO: [CONFIG]
         
T+210ms  主循环开始                           控制循环: CanServicePoll()
         enum_manager_tick()                   → 读 CONFIG → 非枚举帧
         发送 POLL(0x7E3)                     → 存入 s_pendingRx
         发送 OFFER(0x7E0)                    CanServiceReceiveCommand()
                                              → 处理 CONFIG
                                              → ProcessCanCommand()
                                              → DebugUartSendCommandSummary()
                                              → 阻塞 UART 输出 ~5ms！

T+210ms  POLL 到达从机 ────────────────────→  FIFO: [POLL]
         OFFER 到达从机 ───────────────────→  FIFO: [POLL, OFFER]
         
         如有额外命令帧 ───────────────────→  FIFO: [POLL, OFFER, CMD] ← 满！
         再有帧到达 ───────────────────────→  **FIFO OVERRUN! OFFER 被丢弃!**

T+215ms                                     UART 输出完成
                                            控制循环继续
                                            CanServiceReceiveCommand 再调用
                                            → 读 POLL → CanServiceHandleEnum
                                              → node 是 UNSET → 忽略
                                            → FIFO 空（OFFER 已丢失）
                                            → 从机保持 UNSET 状态
```

### 3.2 启动时序竞争

主机启动后需约 200ms 完成 BLE 初始化，然后立即发送 `gateway_broadcast_config()`（CAN CONFIG 帧）并进入主循环。从机的启动时间取决于固件大小和外设初始化顺序。

**时序窗口**：
- 从机 CAN 外设初始化完成的时间 ≈ 80~100ms（从机上电后）
- 主机首帧（CONFIG）发出时间 ≈ 200~300ms（主机上电后）
- 如果主机和从机同时上电，从机约有 **100~200ms 的空闲时间**等待首帧
- 首帧（CONFIG）到达时，恰好落在从机控制循环的某个阶段

**TEST_BUILD 的影响**：测试固件体积更大（多 ~400 行代码 + 额外初始化），启动时间稍有延长，改变了与主机 CONFIG 帧的到达时序关系，使 CONFIG 帧落在控制循环的不同阶段，恰好避免了与 POLL/OFFER 的 FIFO 竞争。

### 3.3 阻塞式 UART 调试输出

#### 影响量化

从机固件中存在多处阻塞式 UART 输出（spin-wait `USART_FLAG_TXBE`）：

| 位置                              | 内容            | 字节数  | 耗时 @115200 |
| ------------------------------- | ------------- | ---- | ---------- |
| `DebugUartSendCommandSummary()` | CAN 命令摘要      | ~55  | ~4.8ms     |
| `DebugPrintMt6701Raw()`         | 传感器原始值        | ~40  | ~3.5ms     |
| `DebugUartSendRuntimeStatus()`  | 运行时状态（每500ms） | ~150 | ~13.0ms    |
| `ProcessCanCommand()` 内各分支      | 命令处理日志        | ~60  | ~5.2ms     |

**关键发现**：`DebugUartSendRuntimeStatus()` 每 500ms 阻塞约 **13ms**，远超一个控制循环周期（10ms）。在此期间，CAN FIFO 完全无人排空，3 帧 FIFO 极易溢出。

#### 代码路径

```c
// main.c → RunControlLoop()
debugPrintTimer += CONTROL_LOOP_MS;
if (debugPrintTimer >= 500U) {
    DebugPrintMt6701Raw();                    // 阻塞 ~3.5ms
    DebugUartSendRuntimeStatus(nowAngle);     // 阻塞 ~13ms
    debugPrintTimer = 0U;
}
```

`ProcessCanCommand()` 中的阻塞输出同样在 CAN FIFO 排空的关键路径上：

```c
// main.c → ProcessCanCommand()
DebugUartSendString("Process CAN: ");
DebugUartSendCommandSummary(cmd);             // 阻塞 ~5ms
// ... switch 分支中还有更多阻塞输出
```

### 3.4 `s_hasPendingRx` 阻塞 FIFO 读取

#### 机理

`CanServicePoll()` 中存在一个微妙的逻辑缺陷 ([can_service.c:384](motor_control_apm32/app/Source/can_service.c#L384))：

```c
void CanServicePoll(void)
{
    CanServiceServiceAck();

    // ⚠️ 当 s_hasPendingRx == 1 时，完全跳过 FIFO 读取！
    if ((s_hasPendingRx == 0U) && 
        (CAN_ReadStatusFlag(CAN1, CAN_FLAG_F0MP) != RESET))
    {
        // 仅读 1 帧
        CAN_RxMessage(CAN1, CAN_RX_FIFO_0, &rx);
        CAN_ReleaseFIFO(CAN1, CAN_RX_FIFO_0);
        // ...
    }
}
```

**场景**：如果上一轮有非枚举帧（如 CONFIG 广播）被缓存为 `s_pendingRx`，本轮 `CanServicePoll()` 完全不读取 FIFO。此时如果 POLL + OFFER 到达，它们必须在 `CanServiceReceiveCommand()` 中排队处理（位于 pending 帧之后），增加了延迟和溢出风险。

### 3.5 CAN 过滤器配置（已排除）

经过对 SPL 驱动源码 ([apm32f10x_can.c](motor_control_apm32/app/Libraries/APM32F10x_StdPeriphDriver/src/apm32f10x_can.c)) 和过滤器配置 ([board_driver.c:59-98](motor_control_apm32/app/Source/board_driver.c#L59-L98)) 的逐行审查：

- **16-bit ID List 模式**：配置正确，Bank 0 覆盖 0x000、0x7DD、0x7E0、0x7E2，Bank 1 覆盖 0x7E3 × 4
- **OFFER（0x7E0）**：`CAN_FILTER16_STD(0x7E0) = 0xFC00`，正确匹配
- **POLL（0x7E3）**：`CAN_FILTER16_STD(0x7E3) = 0xFC60`，正确匹配
- **寄存器写入顺序**：`CAN_ConfigFilter()` 正确操作 FINITEN 进入/退出初始化模式

**结论**：过滤器硬件配置正确，不是根因。

### 3.6 CAN 中断向量表（已排除）

对 GCC 启动文件 ([startup_apm32f103xb.S](motor_control_apm32/app/Libraries/Device/Geehy/APM32F10x/Source/gcc/startup_apm32f103xb.S)) 的审查：

- `USART1_IRQHandler` weak 别名指向 `Default_Handler`（无限循环 `b .`）
- 但 CAN 不使用中断，相关 NVIC 线未使能
- 无 TEST_BUILD 时，USART1 RXBNE 中断也未使能（仅在 `TestShellInit()` 中使能）
- **不存在意外中断触发导致系统挂起的情况**

**结论**：中断向量配置正确，不是根因。

### 3.7 波特率偏差（已排除）

| 节点 | 时钟源 | 波特率配置 | 实际波特率 | 偏差 |
|------|--------|-----------|-----------|------|
| ESP32-C3 | APB 80MHz | TWAI_TIMING_CONFIG_50KBITS | ~49,383 bps | -1.2% |
| APM32F103 | PCLK1 36MHz | BRP=44, TS1=13, TS2=2 | 50,000 bps | 0% |

两节点间波特率偏差约 **1.2%**，在 CAN 协议容忍范围内（每节点 ±1~2%）。且相同硬件在 TEST_BUILD 下工作正常。

**结论**：波特率偏差不是根因。

## 4. 为什么 TEST_BUILD 能"修复"问题

**TEST_BUILD 并非真正修复了问题，而是通过以下副作用改变了运行时序，使 FIFO 溢出窗口恰好被避开：**

| 副作用 | 机制 | 影响 |
|--------|------|------|
| 固件体积增大 | Flash 布局改变、启动时间延长（数ms） | 改变 CONFIG 帧与从机控制循环的相对到达时间 |
| `TestShellInit()` 调用 | 使能 USART1 RX 中断（`USART_INT_RXBNE`），修改 NVIC 和外设状态 | 微秒级中断延迟变化 |
| `TestShellPoll()` 每循环调用 | 在 `CanServiceCheckBusHealth()` 前增加了额外函数调用 | CPU 流水线/缓存状态变化 |
| `ProcessCanCommand` 变为 extern | 编译器无法内联，调用约定改变 | 关键路径执行时间微秒级差异 |

**本质**：这是一个 **Heisenbug**（观测行为改变现象）。TEST_BUILD 带来的微小时序偏移恰好使 CONFIG 帧在从机控制循环中某个"安全"时刻被处理——此时 FIFO 有足够空间容纳后续的 POLL+OFFER 帧。

**警示**：如果在生产固件中改变了任何代码（哪怕是看似无关的修改），时序可能再次变化，导致问题"复发"。必须从根本上修复 FIFO 排空机制，而非依赖时序巧合。

## 5. 进一步确认方案

在实施修复之前，建议通过以下方案确认根因假设。

### 5.1 方案一：硬件 CAN 总线监控（金标准）

**方法**：使用 CAN 逻辑分析仪（如 PEAK PCAN-USB、Kvaser、Saleae + CAN 解码）监控总线，记录帧序列和精确时间戳。

**确认要点**：
- [ ] 确认主机在每个 400ms 周期开始时刻确实发送了 POLL + OFFER
- [ ] 确认 OFFER 帧在 CAN 总线上存在且被 ACK（ACK slot 为显性 = 有节点确认）
- [ ] 检查从机是否发送了 ENUM_ACK（0x7E1）帧
- [ ] 记录从机 ENUM_ACK 帧与 OFFER 帧之间的时间差（确认退避机制正常工作）
- [ ] 检查是否存在 CAN 错误帧（Stuff Error、Form Error 等）

**判断标准**：
- 若 OFFER 帧在总线上存在且被 ACK → 问题在从机软件（FIFO 排空）
- 若 OFFER 帧存在但 ACK slot 为隐性 → 从机 CAN 控制器未正确接收，检查从机侧
- 若 OFFER 帧根本不存在 → 问题在主机侧发送逻辑

### 5.2 方案二：从机侧 CAN 诊断增强

**方法**：在从机固件中添加（非阻塞方式）以下诊断信息：

```c
// 在 CanServicePoll() 或 RunControlLoop() 中采集
struct can_diag {
    uint32_t fifo_overrun_count;   // F0OVR 标志置位次数
    uint32_t rx_error_counter;     // REC 值
    uint32_t tx_error_counter;     // TEC 值
    uint32_t frames_received;      // 软件层面的收帧总数
    uint32_t frames_lost;          // 硬件 FIFO 溢出导致的丢帧估算
    uint32_t enum_poll_count;      // 收到的 POLL 帧数
    uint32_t enum_offer_count;     // 收到的 OFFER 帧数
    uint32_t enum_assign_count;    // 收到的 ASSIGN 帧数
    uint32_t enum_ack_sent_count;  // 发出的 ENUM_ACK 数
};
```

**关键检查点**：
- [ ] `CAN_FLAG_F0OVR`（FIFO0 Overrun Flag）是否被置位
- [ ] `CAN_FLAG_F0FULL`（FIFO0 Full Flag）是否周期性置位
- [ ] 检查 `CAN_ReadRxErrorCounter()` 返回值，确认 REC 是否增长

**注意**：诊断数据必须使用非阻塞方式输出（环形缓冲区 + 中断发送），避免诊断本身影响被观测对象。

### 5.3 方案三：主机侧精确 TX/RX 时间戳日志

**方法**：在 ESP32 主机侧，对每个发送/接收的枚举帧记录微秒级时间戳：

```c
// 在 hal_can_send() 和 hal_can_receive() 中已有时间戳
// 增强：对枚举帧单独记录
if (id == CAN_STD_ID_ENUM_POLL || id == CAN_STD_ID_ENUM_OFFER || 
    id == CAN_STD_ID_ENUM_ASSIGN) {
    ESP_LOGI(TAG, "ENUM TX id=0x%03lx at %lldus", id, esp_timer_get_time());
}
```

**确认要点**：
- [ ] POLL 和 OFFER 是否在同一个微秒窗口内发送（应间隔 < 1ms）
- [ ] ENUM_ACK 的接收时间与 OFFER 发送时间的差值是多少
- [ ] 首个成功的 ENUM_ACK 出现在哪个周期

### 5.4 方案四：控制变量对比测试

## 6. 修复方案

### 6.1 方案A：CAN 中断处理枚举帧（推荐）

**原理**：将从机对枚举帧（POLL/OFFER/ASSIGN）的处理从轮询路径移至 CAN RX 中断服务程序中，实现接近实时的 FIFO 排空。

**修改文件**：
- [can_service.c](motor_control_apm32/app/Source/can_service.c) —— 实现 ISR
- [board_driver.c](motor_control_apm32/app/Source/board_driver.c) —— 使能 CAN 中断和 NVIC
- [can_service.h](motor_control_apm32/app/Include/can_service.h) —— 声明新增接口
- [apm32f10x_int.c](motor_control_apm32/app/Source/apm32f10x_int.c) —— 移除 weak 别名（可选）

**实现要点**：

```c
// ===== can_service.c 新增 =====

// 环形缓冲区，用于从 ISR 向主循环传递非枚举帧
#define CAN_RX_RING_SIZE 8
static CAN_RxMessage_T s_rx_ring[CAN_RX_RING_SIZE];
static volatile uint8_t s_rx_ring_head = 0;  // ISR 写入
static volatile uint8_t s_rx_ring_tail = 0;  // 主循环读取

// CAN1 RX0 中断服务程序
void USBD1_LP_CAN1_RX0_IRQHandler(void)
{
    CAN_RxMessage_T rx;
    
    // 循环排空 FIFO（可能在一次中断中收到多帧）
    while (CAN_ReadStatusFlag(CAN1, CAN_FLAG_F0MP) != RESET)
    {
        CAN_RxMessage(CAN1, CAN_RX_FIFO_0, &rx);
        CAN_ReleaseFIFO(CAN1, CAN_RX_FIFO_0);
        
        // 枚举帧在中断上下文中直接处理（逻辑简单，无阻塞调用）
        if (CanServiceHandleEnum(&rx))
        {
            continue;  // 枚举帧已处理，下一帧
        }
        
        // 非枚举帧存入环形缓冲区
        uint8_t next = (s_rx_ring_head + 1) % CAN_RX_RING_SIZE;
        if (next != s_rx_ring_tail)  // 未满
        {
            s_rx_ring[s_rx_ring_head] = rx;
            s_rx_ring_head = next;
        }
        // 满则丢弃（环形缓冲区足够大时不应发生）
    }
}

// ===== can_service.c 修改 CanServicePoll =====
void CanServicePoll(void)
{
    CanServiceServiceAck();  // 保持不变
    
    // 从环形缓冲区取非枚举帧
    if (s_hasPendingRx == 0U && s_rx_ring_tail != s_rx_ring_head)
    {
        s_pendingRx = s_rx_ring[s_rx_ring_tail];
        s_rx_ring_tail = (s_rx_ring_tail + 1) % CAN_RX_RING_SIZE;
        s_hasPendingRx = 1U;
    }
    // 删除原有的直接 FIFO 读取代码
}

// ===== board_driver.c 修改 BoardCanInit =====
void BoardCanInit(void)
{
    // ... 原有初始化代码 ...
    
    // 新增：使能 FIFO0 消息挂起中断
    CAN_EnableInterrupt(CAN1, CAN_INT_F0MP);
    NVIC_EnableIRQ(USBD1_LP_CAN1_RX0_IRQn);
}
```

**优点**：
- 彻底消除 FIFO 溢出风险（ISR 响应时间 < 10μs）
- 枚举帧处理延迟从 10ms 降至微秒级
- 不依赖时序巧合，在任何固件配置下都可靠

**缺点**：
- 改动较大，需充分测试
- ISR 中不能有阻塞调用（当前 `CanServiceHandleEnum` 无阻塞调用，符合要求）
- 需考虑 ISR 与主循环之间的数据竞争（环形缓冲区需正确设计）

**工作量**：中等（约 4 个文件，~100 行新增代码）

### 6.2 方案B：CanServicePoll 循环排空（最小改动）

**原理**：将 `CanServicePoll()` 中对 FIFO 的单帧读取改为循环排空，确保每次调用排空所有可用帧。

**修改文件**：[can_service.c](motor_control_apm32/app/Source/can_service.c) 仅 1 处修改。

**修改内容**（[can_service.c:379-397](motor_control_apm32/app/Source/can_service.c#L379-L397)）：

```c
void CanServicePoll(void)
{
    CanServiceServiceAck();

    // 修改前：if ((s_hasPendingRx == 0U) && (...))  ← 单帧读取
    // 修改后：while 循环排空所有枚举帧
    while ((s_hasPendingRx == 0U) && 
           (CAN_ReadStatusFlag(CAN1, CAN_FLAG_F0MP) != RESET))
    {
        CAN_RxMessage_T rx;
        CAN_RxMessage(CAN1, CAN_RX_FIFO_0, &rx);
        CAN_ReleaseFIFO(CAN1, CAN_RX_FIFO_0);

        if (!CanServiceHandleEnum(&rx))
        {
            s_pendingRx = rx;
            s_hasPendingRx = 1U;
            break;  // 非枚举帧存入pending，留给CanServiceReceiveCommand
        }
    }
}
```

**优点**：
- **改动仅 1 个字符**（`if` → `while`）
- 同一控制循环内排空所有枚举帧
- 几乎零风险

**缺点**：
- 仍受 10ms 轮询周期限制（FIFO 在两次轮询之间仍可能溢出）
- 阻塞式 UART 输出期间 CAN 帧仍会丢失
- 是缓解措施，不是根治

**工作量**：极小（1 个文件，1 处修改）

### 6.3 方案C：非阻塞 UART 输出

**原理**：将阻塞式 UART 输出改为中断驱动的环形缓冲区 + DMA 发送，消除对 CAN FIFO 排空的阻塞影响。

**修改文件**：[main.c](motor_control_apm32/app/Source/main.c) 中的 Debug UART 函数族。

**实现要点**：

```c
// 环形缓冲区（替代直接 spin-wait）
#define UART_TX_RING_SIZE 256
static char s_tx_ring[UART_TX_RING_SIZE];
static volatile uint16_t s_tx_head = 0;  // 生产者（main loop）
static volatile uint16_t s_tx_tail = 0;  // 消费者（ISR）

// 非阻塞 putc
static void DebugUartSendChar(char ch)
{
    uint16_t next = (s_tx_head + 1) % UART_TX_RING_SIZE;
    while (next == s_tx_tail) { /* 等待空间（几乎不会等） */ }
    s_tx_ring[s_tx_head] = ch;
    s_tx_head = next;
    USART_EnableInterrupt(USART1, USART_INT_TXBE);  // 启动发送
}

// TX 中断服务程序
void USART1_IRQHandler(void)
{
    if (USART_ReadIntFlag(USART1, USART_INT_TXBE) != RESET)
    {
        if (s_tx_tail != s_tx_head)
        {
            USART_TxData(USART1, s_tx_ring[s_tx_tail]);
            s_tx_tail = (s_tx_tail + 1) % UART_TX_RING_SIZE;
        }
        else
        {
            USART_DisableInterrupt(USART1, USART_INT_TXBE);  // 发送完毕
        }
    }
    // 如果有 TEST_BUILD，还需处理 RXBNE（合并两个 ISR）
}
```

**优点**：
- 消除 CAN 排空路径上的所有阻塞
- 改善整体系统响应性

**缺点**：
- 需合并 `USART1_IRQHandler`（当 TEST_BUILD 启用时已有 RX ISR）
- 环形缓冲区满时需处理（丢弃或等待）

**工作量**：中等

### 6.4 方案D：增大主机收集窗口

**原理**：将主机的 `ENUM_COLLECT_WINDOW_MS` 从 250ms 增大到 500ms，给从机更长的响应时间。

**修改文件**：[enum_manager.c](ble_esp32/main/app/enum_manager.c) 第 12 行。

```c
#define ENUM_COLLECT_WINDOW_MS  500U  // 原值 250U
```

**优点**：改动极小

**缺点**：
- 不解决帧丢失问题，只是增加了"等待时间"
- 增加了枚举收敛的延迟
- 从机若持续丢帧，窗口增大也无用

|------|:--:|:--:|:--:|:--:|

## 7. 风险评估

### 7.1 当前影响

| 维度 | 评估 |
|------|------|
| **功能影响** | 中等——枚举最终能收敛（多个cycle后），不阻塞正常使用，但增加了上电到就绪的延迟 |
| **多节点影响** | 高——多从机（2~6个）场景下，FIFO 溢出概率显著增加（总线帧密度 × N），可能导致某些节点反复无法枚举 |
| **可靠性** | 高——依赖时序巧合而非确定性机制，任何固件变更都可能使问题复发或恶化 |
| **生产部署风险** | 高——生产固件（无 TEST_BUILD）恰好暴露此问题，出厂设备的枚举稳定性无法保证 |

### 7.2 长期风险

1. **维护陷阱**：任何对 `main.c`、`can_service.c` 的修改都可能改变时序，导致"已修复"的问题复发
2. **扩展性**：增加从机数量或增加 CAN 总线负载（如新增协议帧）会显著提高 FIFO 溢出概率
3. **耐久性**：老化测试中可能暴露更多偶发失效（总线错误率随时间增加）

### 7.3 建议

**强烈建议实施方案A（CAN中断处理枚举帧）**，从根本上消除对时序的依赖。这不仅是修复当前问题，更是对系统架构的正确改进——枚举帧属于实时性要求高的控制帧，理应在中断上下文中处理。
