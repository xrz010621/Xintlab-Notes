| 3   | 连续观察 5 个 cycle（约 3 秒） | 每个 cycle 都有 `RX POLL, myId=1, will ACK` + `TX ACK id=1`，无中断                                      | —            |

### 4.2 3 节点联动测试

## 五、Phase 2 — 反复上电耐久测试（核心复现手段）

### 5.1 测试逻辑

```
┌──────────────────────────────────────────────────────────┐
│                 一轮测试 (1 trial)                         │
│                                                          │
│  1. 关排插 → 主控+3从机统一断电                            │
│  2. 等待约 3 秒（电容放完、所有板彻底复位）                  │
│  3. 开排插 → 统一上电                                      │
│  4. 等待约 10 秒（观察 3 路串口终端，确认枚举完成）          │
│  5. 快速判定本轮结果：                                     │
│     ├── PASS: 3块从机各自日志显示完整 RX OFFER → TX ACK    │
│     │         序列，各自拿到不同 ID，稳态无异常              │
│     ├── FAIL_DELAYED: 某块从机出现 intermittent 缺失       │
│     │       但最终拿到 ID（如需要额外 ASSIGN）              │
│     ├── FAIL_INCOMPLETE: 某块从机 10秒后仍只有              │
│     │       RX POLL skip UNSET，从未出现 RX OFFER          │
│     └── FAIL_STUCK: 某块从机的日志突然完全中断              │
│             （多秒无任何 RX POLL，疑似 CAN 外设被复位）     │
│  6. 保存本轮日志文件                                       │
└──────────────────────────────────────────────────────────┘
```

### 5.2 具体操作步骤

采用**手动方式**，不依赖继电器或程控电源：

1. 主控 + 3 从机插在同一个带开关的排插上
2. 上位机打开 3 个串口终端（方式 A：PuTTY / Tera Term），各自配置 115200 8N1 并开启 log-to-file
3. 手动操作：**关排插 → 默数 3 秒 → 开排插 → 观察约 10 秒 → 关排插**
4. 每轮结束后在终端日志中标记序号和结果（如输入注释 `=== trial 012 PASS ===`），然后将日志文件另存到 `test_results/trial_NNN/` 目录
5. 重复

**每一轮的实际耗时**：断电 3s + 观察 10s + 文件操作 2s ≈ **15 秒/轮**。手动 50 轮约需 13 分钟。

当进入批量阶段（50+ 轮）时，可将方式 A 切换为方式 B（Python 脚本合并采集），减少每轮的文件操作开销。

### 5.3 测试规模设计

| 阶段 | 轮数 | 累计耗时 | 目的 |
|------|------|---------|------|
| 摸底 | 50 | ~13 分钟 | 初步判断异常率数量级 |
| 若 50 轮内抓到异常 | 追加到 100~200 | 25~50 分钟 | 获得更准确的异常率估计 |
| 若 50 轮全 PASS | 酌情追加或进入 Phase 3 加速 | — | 判定是否需引入参数压缩 |

> 手动测试的实操上限约 200 轮（50 分钟）。如果异常率低于 1/200，手动方式无法捕获，需后续引入自动化上下电或 Phase 3 参数压缩加速。

### 5.4 判定标准（从机视角，不依赖主控日志）

**PASS**：
- 上电后 2 秒内，3 块从机日志各自出现 `RX OFFER id=N, accept, ack pending`
- 各自 `TX ACK id=N` 紧随其后出现
- 后续 cycle 中每块从机的 `myId` 稳定不变，持续规律输出 `RX POLL + TX ACK`

**FAIL_DELAYED**（轻度异常 — 自愈成功）：
- 某块从机连续 ≥2 个 cycle 仅出现 `RX POLL skip UNSET`（无 `RX OFFER`），但后续恢复并最终拿到 ID
- 某块从机收到 `RX ASSIGN match accept`（被重定位到新 ID），但最终收敛

**FAIL_INCOMPLETE**（中度异常 — 自愈失败）：
- 上电 10 秒后，某块从机日志只有 `RX POLL skip UNSET`，从未出现 `RX OFFER accept`
- 某块从机日志中最后一条 `RX POLL` 之后多秒再无任何输出

**FAIL_STUCK**（重度异常 — CAN 通信中断）：
- 某块从机的日志突然完全中断（无任何 RX 输出、无 TX ACK），多秒未恢复
- 需对该从机单独重新上电才能恢复

### 5.5 日志保存策略

```
test_results/
├── trial_001/
│   ├── slave_A.log      ← 通过 UID 前缀区分实际对应哪块板
│   ├── slave_B.log
│   ├── slave_C.log
│   └── verdict.txt      ← 一行: PASS / FAIL_xxx + 关键摘要
├── trial_002/
...
└── summary.csv          ← trial编号, 结果, 异常从机UID, 异常类型, 备注
```

## 六、Phase 3 — 故障注入与加速暴露

如果 Phase 2 的异常率太低（50 轮全 PASS），通过以下手段提升触发概率。当前阶段仅关注**软件层面**的可调参数，不涉及硬件改动。

### 6.1 参数边界压缩

**操作**：在从机 `app_config.h` 中临时修改编译参数，缩小安全余量。重新编译诊断固件后烧录。

| 参数 | 正常值 | 测试值 | 目的 |
|------|--------|--------|------|
| `NODE_ID_ENUM_ACK_SPREAD_MS` | 48 ms | 16 ms | 缩小退避窗口，增加 ACK 碰撞概率 |

> 注意：`ENUM_COLLECT_WINDOW_MS`（250ms）和 `ENUM_PERIOD_ACTIVE_MS`（400ms）定义在 ESP32 主控侧（`enum_manager.c`），主控固件本轮不修改，因此**暂不可调**。

**警告**：这些值仅用于测试，不得合入正式固件。测试完成后恢复。

### 6.2 增加节点数

**操作**：将测试节点从 3 块恢复到 6 块。

**目的**：更多节点 = 更多 ACK 集中在 250ms 窗口内 = 更高的碰撞/仲裁丢失概率。

**代价**：需 6 路 CH340，日志采集路数翻倍，操作复杂度增加。优先在 3 块条件下先跑 Phase 2。

## 七、Phase 4 — 正常运行时的影响评估

**前提**：Phase 2/3 已确认根因与 CAN 帧丢帧相关。

### 7.1 测试逻辑

如果枚举阶段的根因是"CAN 总线在特定条件下可能丢帧"，需要评估该丢帧是否也影响正常运行时的电机指令帧。

1. 3 块从机枚举完成后，通过 BLE / Web 前端发送运动指令（如 `127,127,127,-1,-1,-1`，50% 高度）
2. 在从机日志中观察：当前 `main.c` 已有 `DebugUartSendCommandSummary` 输出每条收到的 CAN 指令（`ProcessCanCommand`），可通过对比"前端发送次数"和"从机日志中 `CAN action=control` 的出现次数"来判断指令帧是否丢帧
3. 连续运行 1 小时，统计正常运行阶段的指令漏收率，与枚举阶段的 OFFER 漏收率对比

### 7.2 判定

- 若正常运行丢帧率显著低于枚举阶段 → 问题仅限于枚举突发帧密度
- 若正常运行也有类似丢帧率 → **这是一个系统级 CAN 可靠性问题，不只是枚举问题**

## 八、Phase 5 — 根因判定矩阵

将 Phase 2~4 收集的现象对号入座：

| 观察到的日志特征 | 根因方向 | 验证手段 |
|-----------------|---------|---------|
| 某块从机连续多 cycle 仅 `RX POLL skip UNSET`，从不出现 `RX OFFER` | **从机 CAN RX 丢帧**（OFFER 帧未到达） | 检查该从机 CAN RX FIFO 溢出标志、UART 打印时序是否干扰了 CAN 读取节奏 |
| 从机日志显示 `TX ACK id=N`，但后续 cycle 该从机收到 `RX ASSIGN match accept`（主控认为它没回应） | **总线碰撞/仲裁丢失**（ACK 在线上丢了，主控判为冲突并重定位） | 对照其他从机日志中的 ASSIGN target，确认哪些 UID 被重定位 |
| 某块从机的日志突然完全中断（无任何 RX 输出），持续多秒 | **从机 CAN 外设 Error Passive 被静默复位** | 后续启动 CAN 错误日志功能，在 `CanServiceCheckBusHealth` 复位前打印 TEC/REC |
| `TX ACK` 出现但明显少于 `RX POLL will ACK` 的次数 | **ACK 槽位被覆盖**（单槽设计：POLL 的 ACK 被同 tick 的 OFFER/ASSIGN 覆盖） | 检查日志中 OFFER/ASSIGN 是否与 POLL 在同一 tick 到达 |
| 问题随 48ms 退避缩短后显著恶化 | **退避不足**（`ACK_SPREAD_MS` 对 3~6 节点不够） | 理论计算 N 节点 × 48ms 的碰撞概率；若确认则增大 `ACK_SPREAD_MS` |

## 十、方案评估

### 10.1 优势

1. **改动范围极小**：主控完全不动，从机侧采用"增添而非修改"策略，`enum_diag.c/h` 独立于已有代码，编译开关控制，不影响正式固件
2. **日志即证据**：每个枚举帧的收/发都在串口留有文本行，异常时直接定位断裂点——是没收到 OFFER 还是发了 ACK 主控没收到，不需要推测
3. **零硬件门槛启动**：Phase 1~2 只依赖排插 + 3 个 CH340 + 串口终端软件，不需要继电器、程控电源或其他特殊设备
4. **渐进投入**：Phase 1 确认环境 → Phase 2 摸底 50 轮 → 根据结果决定是否追加或进入 Phase 3，不盲目投入
5. **判定标准独立于主控**：不再依赖主控侧的 `assigned=6` 确认枚举完成，改为从机日志自证——"我收到了 OFFER、我发出了 ACK、我的 ID 稳定了"

### 10.2 局限与风险

1. **诊断日志本身影响时序**：`EnumDiagSendChar` 约 87μs/字符 × 60 字符/行 ≈ 5ms/行。在枚举密集期（POLL+OFFER 同 tick）约 10ms 额外 CPU 阻塞。缓解：`ENUM_DIAG` 仅测试固件使用，正式固件不受影响；枚举阶段不走电机闭环，延迟可接受
2. **手动测试的规模上限**：15 秒/轮 × 200 轮 = 50 分钟人工盯屏，这是实操上限。若异常率低于 1/200，手动无法捕获。缓解：50 轮摸底后若全 PASS，进入 Phase 3 参数压缩加速；或后续引入继电器自动上下电
3. **只有从机视角**：无法直接确认主控侧是否收到 ACK。从机日志显示 `TX ACK` 但主控因总线碰撞没收到 → 下一 cycle 可能以 `RX ASSIGN` 呈现。从机日志可以**间接推断**主控侧行为，但无法直接确认
4. **CAN 错误日志暂不启用**：若根因是 CAN 物理层错误累积导致静默复位，Phase 2 的日志可能只能看到"日志突然中断"，无法直接确认是复位。如需定位此方向，需在 Phase 2 异常发生后临时启用 `EnumDiagLogCanError` 重新测试
5. **缺少"修复后回归"的明确定义**：本方案聚焦于复现和定位，修复方案取决于根因，回归测试的验收标准需等根因确定后再制定

### 10.3 总体评价

本方案是一个**以从机侧诊断日志为核心、以手动可量化复现为目标、以软件层面渐进式排查为节奏**的测试框架。它不依赖额外硬件设备，不修改主控固件，每一步都产出可验证的事实。最大的投入在 Phase 2（手动耐久测试），而其产出（日志判据 + 测试脚本）本身也是长期可复用的资产。
